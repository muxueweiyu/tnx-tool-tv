# 新增功能标准化工作流 (build-feature.md)

当用户让你“增加一个新功能” 或 “对接一个新的服务平台”时，AI 助手**必须自动、强制触发以下步骤，绝不能在一个文件内强行全部写完**：

## 步骤 1：确立数据模型边界 (Models)
在写任何执行代码前，先思考：这个新功能需要输入什么数据？会吐出什么数据？
- 行动：在 `src/core/models/` 目录下创建或修改相关的 TypeScript Interfaces。
- 目的：斩断隐形依赖，用强类型切断不同模块传递混乱字典的可能。

## 步骤 2：创建隔离的纯服务机制 (Services & Infrastructure)
- 如果这个功能要进行网络连接、读写本地文件？
  - 行动：先在 `src/infrastructure/` 下建一个新文件（比如 `api_client.ts` 或 `file_reader.ts`），只包装对外界的纯粹交互。
- 如果是核心运算、数据组装？
  - 行动：在 `src/services/` 下新建一个处理该逻辑的专职文件。注入它所需要的 infrastructure。

## 步骤 3：命令层接入 (Commands/CLI)
在确认核心逻辑已经解耦编写完成后。
- 行动：去主路由处（例如建一个 `src/commands/my_new_cmd.ts`），把用户的输入解析为第 1 步中的数据模型，然后将数据丢给第 2 步的服务即可。

### 🚨 AI 强制自检 🚨
提交流程前，请 AI 自己检查：
- [ ] 这个新增的代码是否往原有文件中硬塞了超过 20 行的 `if` 逻辑？若是，改用抽象工厂或接口实现新建文件！
- [ ] 核心 API 报错时，`commands` 会直接崩溃吗，还是被 `services` 抓获并返回了标准化错误对象？
